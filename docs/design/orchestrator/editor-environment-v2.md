# Editor Redesign Proposal

## Overview

Replace the current dual-pattern editor with a pipeline-based query system. All views are generated procedurally from queries that re-execute on every screen refresh, ensuring screens are never stale even as files change.

## Core Principle: Procedural Views

**All views are queries, not snapshots.**

When files change, the screen regenerates by re-executing all view queries. This ensures the screen always shows current content at the correct locations.

```
Agent: view /def process_data/ in file.py | while-indent
→ Stores QUERY, not line numbers

File edited, lines shift
→ Screen regeneration re-executes query
→ Finds /def process_data/ at NEW location
→ Expands from current position
→ Screen shows correct content
```

This is why we CANNOT support direct line number access - line numbers become stale immediately.

## Core Concepts

### Windows
A **window** is a contiguous range of lines from a single file, generated by executing a query.
- Has: file path, start line, end line, content, label
- Generated fresh on each screen refresh
- May overlap or duplicate with other windows

### Views
A **view** is what appears on screen after merging overlaps and deduplication.
- Created from windows after merge/dedup
- Has: file path, line range, label(s)
- Regenerated on every screen refresh

### Query Pipelines
Queries are procedures that generate windows through composable operations.

## Commands

### Query Commands

**`view <label> <matcher> in <glob> | <operations>`** - Labeled persistent views
```bash
view secrets /sops\.secrets/ in *.nix | context 5
view network_cfg /# Network/ in containers.nix | until-blank
```
- Label is **mandatory** (first parameter)
- Label naming: alphanumeric, underscore, hyphen (no spaces)
- Same label **overrides** previous query (replaces it)

**`peek <matcher> in <glob> | <operations>`** - Transient reads (no label)
```bash
# Pattern matching
peek /def \w+/ in lib/generation.nix | context 2

# Line matching (only available in peek)
peek line 127 in src/main.rs | context 5
peek line 120-135 in src/main.rs
```

Note: Line matchers only available in `peek` because they're for targeted lookups based on external info (compiler errors, test failures, etc.). Persistent views must use pattern matchers to stay up-to-date as files change.

### View Management Commands

**`close label <string>`** - Remove query with this label
```bash
close label secrets
```

**`close pattern <glob>`** - Remove all queries matching label pattern
```bash
close pattern "hypothesis_1_*"
# Removes hypothesis_1_credentials, hypothesis_1_secrets, etc.
```

**`close file <path>`** - Exclude file from all existing queries
```bash
close file containers.nix
# Modifies all active queries: *.nix becomes *.nix (excluding containers.nix)
# New queries can still match this file
```

**`close all`** - Remove all queries
```bash
close all
```

## Pipeline Operations

**Operation Order:** Operations are applied left-to-right in the pipeline. The logical sequence is: matcher → expansion operations → filtering operations → limit. This ordering ensures operations have the data they need (e.g., expand before filter, filter before limit).

### Matchers (produce initial windows)

**Pattern matcher: `<pattern> in <glob>`**
- Pattern: Python regex
- Glob: File patterns (supports *, **, etc.)
- Returns: One single-line window per match
- Available in: `view` and `peek`

```bash
/TODO/ in src/**/*.py
# Finds all lines containing "TODO" in Python files under src/
```

**Line matcher: `line N` or `line N-M` in `<file>`**
- N: Specific line number (1-based)
- N-M: Line range (inclusive)
- Returns: One window containing specified lines
- Available in: `peek` ONLY (not `view`, since line numbers become stale)

```bash
peek line 127 in src/main.rs | context 5
# Shows line 127 ± 5 lines context

peek line 120-135 in src/main.rs
# Shows lines 120-135

peek line 130 in file.c | up-until /^\w+.*{/ | until /^}/
# Line 130, expand to containing function
```

**Why line matchers only in peek:**
External tools (compilers, linters, test runners) report errors at specific line numbers. Line-based peek allows quick inspection of those locations. But persistent views can't use line numbers because they'd show wrong content after file edits.

### Expansion Operations

**`context <n>`** - Expand n lines up and down
```bash
/def main/ in *.py | context 10
# Shows 10 lines above and below each match
```

**`up <n>`** - Expand n lines upward only
**`down <n>`** - Expand n lines downward only

**`until <pattern>`** - Expand down until pattern matches (alias: `down-until` for explicitness)
```bash
/^def / in *.py | until /^def |^class /
# Expands from "def" line until next function or class
```
- Stops when pattern matches (match line NOT included)
- Max expansion: 200 lines
- If pattern never matches: stops at max expansion or end of file

**`up-until <pattern>`** - Expand up until pattern matches
```bash
peek line 130 in file.c | up-until /^\w+.*{/ | until /^}/
# From line 130, find function start upward, then closing brace downward
```
- Expands upward until pattern matches
- Match line IS included (becomes new window start)
- Max expansion: 200 lines
- If pattern never matches: stops at max expansion or beginning of file

**`until-blank`** - Expand down until blank line
```bash
/^## / in *.md | until-blank
# Markdown sections: expands until empty line
```
- Stops at first completely blank line (blank line NOT included)
- Max expansion: 200 lines
- If no blank line found: stops at max expansion or end of file

**`while-indent`** - Expand while indented relative to window start
```bash
/^def / in *.py | while-indent
# Python function: expands while indented deeper than "def" line
```

**How `while-indent` works:**

1. **Reference indentation:** First line of current window
2. **Include lines:** While indentation > reference indentation
3. **Empty/whitespace-only lines:** Treated as indent > reference indentation
   - Blank lines (no content)
   - Lines with only spaces/tabs
   - This means blank lines (e.g. within Python functions) will NOT stop expansion.

4. **Smart closing:** After stopping, includes ONE more line if it matches `^\s*[\])}>]\s*` (single closing brace/bracket/paren)
   - Only ONE line is auto-included
   - Only if the ONLY non-whitespace content is a closing character
   - This handles C-style `}` or `});` lines

**Example - Python function:**
```python
def func():          # Line 2, indent=4, window starts here
    for i in range(10):  # indent=8 > 4, include
        print(i)     # indent=12 > 4, include
    print("done")    # indent=8 > 4, include
                     # empty line, included
def outer():         # Stop
```

Result: Lines 2-5 (complete function body)

**Example - Nested Python:**
```python
def outer():
    def func():      # Line 2, indent=4, window starts here (after up-until)
        for i in range(10):
            print(i) # Line 4 - where peek started
        print("done")
    func()           # indent=4, NOT > 4, STOP
```

Query: `peek line 4 in file.py | up-until /def/ | while-indent`
- Start at line 4
- `up-until /def/` finds line 2: `def func():`
- Window now starts at line 2 (indent=4)
- `while-indent` expands: includes lines with indent > 4
- Stops at line 6 (`func()`) which has indent=4 (NOT > 4)
- Result: Lines 2-5 (the `func()` function, not `outer()`)

**Example - C function with braces:**
```c
void process() {     # Line 10, indent=0, window starts here
    int x;           # indent=4 > 0, include
    return x;        # indent=4 > 0, include
}                    # indent=0, NOT > 0, but matches ^\s*}, AUTO-INCLUDE
                     # empty line, STOP
```

Result: Lines 10-13 (complete function including closing brace)

**Language-specific usage:**

Python:
```bash
/^def / in *.py | while-indent
/^class / in *.py | while-indent
```

C/Rust/JavaScript (braces):
```bash
/^\w+.*{/ in *.c | while-indent
# Auto-includes closing brace due to smart closing rule
```

Nested structures:
```bash
peek line 47 in nested.py | up-until /^def |^class / | while-indent
# Find enclosing function/class, show complete block
```

### Filtering Operations

**`filter <pattern>`** - Keep only windows containing pattern
```bash
/my_var/ in *.c | context 3 | filter /your_var/
# Find my_var with context, keep only if your_var also appears
```

**`exclude <pattern>`** - Remove windows containing pattern
```bash
/TODO/ in **/*.py | exclude /LATER/
# Find TODOs but exclude ones marked LATER
```

**`limit <n>`** - Keep only first n windows
```bash
/error/ in *.log | context 5 | limit 10
# First 10 errors only
```



## View Creation Flow

### Phase 1: Pipeline Execution
```
1. Parse query string → pipeline AST
2. Execute search (ripgrep backend)
3. Apply transformations sequentially:
   - Expansion operations
   - Filtering operations
   - Limit operations
4. Result: Set of windows
```

### Phase 2: Size Check
```
total_lines = sum(window.line_count for window in windows)

For peek commands:
    if total_lines > PEEK_HARD_LIMIT (300):
        Error: "Query matched X lines - exceeds limit. Refine query."
        STOP

For view commands:
    # Check total with all existing queries
    total_with_existing = total_lines + sum(all existing query lines)
    if total_with_existing > MAX_TOTAL_LINES (3000):
        Error: "Would exceed total limit (X/3000 lines). Close some views first, or refine query."
        STOP
```

### Phase 3: AI Summary Generation

```
# Summary is generated for ALL commands in one <editor>...</editor> tag

On editor tag close:
1. Collect all windows from ALL commands in the tag (view + peek)
2. Create summary prompt:
   "Summarize these code sections in 2-3 sentences.
    Focus on: <inferred from patterns in all commands>

    [all window contents from all commands]

    Format: Clear sentences explaining what this code does."

3. Execute single-turn LLM conversation (separate from main conversation)
4. Summary appears in tool output once at end of tag
5. Log summary generation (tokens, cost)
```

Note: One summary per `<editor>...</editor>` tag, covering ALL windows opened within it.

### Phase 4: Store Query and Check for Auto-Removal

```
For view commands only:
    # Remove old query with same label (override)
    if label in active_queries:
        remove active_queries[label]

    # Check if new query has any windows
    if windows is empty:
        # Query matched nothing - don't add it
        # (This is auto-cleanup for exhausted queries)
        return

    # Store query with metadata:
    - Label (mandatory)
    - Original query string
    - Glob pattern (may be modified by close file commands)

Purpose: Re-execute query on every screen refresh
```

### Phase 5: Add to Active Query Set

```
active_queries[label] = query

# Active queries are re-executed on each screen refresh
# This ensures views stay up-to-date as files change
```

### Phase 6: Execute All Queries & Generate Screen

```
# This happens on EVERY screen refresh
all_windows = []
queries_to_remove = []

for label, query in active_queries.items():
    windows = execute_query(query)

    # Auto-remove queries that match nothing
    if windows is empty:
        queries_to_remove.append(label)
        continue

    all_windows.extend(windows)

# Remove exhausted queries
for label in queries_to_remove:
    del active_queries[label]

# Merge overlapping windows within each file
for file in grouped_by_file(all_windows):
    merge_overlapping_windows(file)

# Deduplicate
all_windows = deduplicate(all_windows)

# Generate screen display
render_views(all_windows)
```

## View Management

### Active Query Set
- Maintains all view queries indexed by label
- Each query re-executes on screen refresh
- Each query has: label (key), query string, glob pattern

### Query Removal

Queries are removed from active set when:

1. **Explicit close:** `close label <name>`
2. **No matches:** Query returns 0 windows (auto-removed on screen refresh)
3. **Label override:** New `view` command with same label replaces old query

### File Exclusion

`close file <path>` modifies ALL existing queries:
```
Before: view secrets /sops/ in **/*.nix
After:  view secrets /sops/ in **/*.nix (excluding containers.nix)

# New queries are unaffected
view new /pattern/ in **/*.nix  # Can still match containers.nix
```

Exclusions are part of query state and persist across screen regenerations.

**Note on query override:** When a query is overridden (same label used again), exclusions do not carry over. The new query uses its specified glob pattern without modifications from previous exclusions.

## Edit Operations

```
edit <file> <start>-<end>
<new content on following lines>
```

Verification:
1. Find view(s) containing lines
2. Verify lines are visible in a view
3. Verify content matches current view content
4. Perform edit
5. Trigger screen regeneration (re-execute all queries)
6. Mark query as "used" (update timestamp)

After edit, screen shows updated content at potentially new line numbers.

## Implementation Architecture

### Components

```
orchestrator/environments/
  editor.py                # Main environment
  editor/
    __init__.py
    parser.py              # Query parser (pyparsing)
    executor.py            # Pipeline executor (ripgrep backend)
    windows.py             # Window/view management, merge/dedup
    summarizer.py          # AI summary generation
    indentation.py         # Indentation analysis for while-indent
```

### Query Execution Engine

```python
class QueryExecutor:
    def execute(self, query_ast):
        # 1. Search operation
        windows = self.search(query_ast.pattern, query_ast.glob)

        # 2. Apply transformations
        for operation in query_ast.operations:
            if operation.type == "context":
                windows = self.expand_context(windows, operation.n)
            elif operation.type == "while-indent":
                windows = self.expand_while_indented(windows)
            elif operation.type == "until":
                windows = self.expand_until(windows, operation.pattern)
            # ... other operations

        return windows
```

### Screen Regeneration

```python
def regenerate_screen():
    all_windows = []

    # Re-execute all active queries
    for query in active_queries:
        windows = executor.execute(query.ast)
        all_windows.extend(windows)

    # Merge and deduplicate
    views = merge_and_dedup(all_windows)

    # Render
    return render_screen(views)
```

## Design Rationale

### Why Procedural Views?

**Problem with snapshots:**
```
Agent creates view at lines 100-150
Agent edits lines 50-60, adds 20 lines
View shows lines 100-150 (WRONG CONTENT - stale)
```

**Solution with queries:**
```
Agent creates: view /def process/ in file.py | while-indent
Agent edits earlier in file
Screen refresh re-executes query
View shows def process at NEW location (CORRECT)
```

### Why One while-indent Is Enough

**Smart closing rule** handles different languages automatically:

**Python:**
```python
def foo():        # indent 0, start
    x = 1         # indent 4 > 0, include
    return x      # indent 4 > 0, include
                  # empty (indent -1, NOT > 0), STOP
```

**C (auto-includes closing brace):**
```c
void foo() {      # indent 0, start
    int x;        # indent 4 > 0, include
    return x;     # indent 4 > 0, include
}                 # indent 0, NOT > 0, BUT matches ^\s*}, auto-include
                  # empty, STOP
```

One operator works for both because closing braces are special-cased.

### Why Not Tree-Sitter?

Tree-sitter would be ideal for semantic expansion, but:
- Requires language-specific parsers (50+ languages)
- Adds significant complexity
- Indentation-based expansion handles 80% of cases
- Can add tree-sitter later as enhancement via `expand-block` operation

For now, pattern + indentation is simpler and works for most cases.
